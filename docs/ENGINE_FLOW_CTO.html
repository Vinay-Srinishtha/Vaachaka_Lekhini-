1. Overview

Voice Engine

Captures PCM audio, sends it to Vosk, receives partial/final text, and counts only when the complete expected mantra phrase appears in the recognized output.

Writing Engine

Captures on-screen strokes and PNG output. English can use image OCR, while Indic scripts prefer stroke-based Digital Ink recognition. Both paths return the same HandwritingResult.

Shared Session Layer

Voice, writing, and manual counts all become local sessions. Session finish updates program totals, streaks, and queues backend sync through the outbox.

4. Voice Engine Flow

4.5 Continuous Chanting Handling

Vosk usually finalizes when it detects silence. Continuous chanting may not provide clean silence, so the service uses timed finalization:

  • Poll every 300 ms.
  • Finalize after approximately 180 ms of silence.
  • Force finalize after 2000 ms if there is no gap.
  • Buffer chunks while finalizing so the next chant is not dropped.

5. Writing Engine Flow

The writing flow exports both PNG bytes and stroke points. It chooses the recognition path based on the available input and script, then collapses both paths into one shared result object.

8. Code-Level Contracts

These are the minimum code-level snippets needed to explain how the implementation actually moves data between modules. They are intentionally focused on the contracts, not the entire source files.

8.1 Audio Capture Contract

The voice engine depends on a live PCM stream. The app uses record to request microphone permission and emit 16 kHz mono PCM16 chunks. 

class AudioCapture {
  static const int sampleRate = 16000;

  Future<Stream<Uint8List>> start() async {
    if (!await ensurePermission()) {
      throw StateError('Microphone permission denied');
    }

    final stream = await _recorder.startStream(const RecordConfig(
      encoder: AudioEncoder.pcm16bits,
      sampleRate: sampleRate,
      numChannels: 1,
      noiseSuppress: false,
      echoCancel: false,
    ));

    return stream;
  }
}

8.2 Vosk Recognizer Contract

Vosk is wrapped so the rest of the app receives one consistent object: RecognitionResult(text, words, isFinal). 

class RecognitionResult {
  final String text;
  final List<WordResult> words;
  final bool isFinal;
}

Future<RecognitionResult?> acceptChunk(Uint8List pcm16) async {
  final isFinal = await recognizer.acceptWaveformBytes(pcm16);
  final raw = isFinal
      ? await recognizer.getFinalResult()
      : await recognizer.getPartialResult();

  return parseVoskJson(raw, isFinal);
}

10. Readiness Notes

  • Local-first recognition: Voice and handwriting recognition run on-device in the current app path.
  • Recognition boundary: Decisions are based on the text returned by the recognition engine.
  • User recovery: Missed voice recognition, unreadable writing, and denied mic access leave counts unchanged and allow retry or manual fallback.
  • Durability: Practice deltas flush periodically and drafts preserve active session count across app interruption.
  • Auditability: Finished sessions record member, program, modality, count, start/end, and duration.
  • Sync shape: Backend receives compact session/program payloads, not raw audio or raw handwriting strokes.
